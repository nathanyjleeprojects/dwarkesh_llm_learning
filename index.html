    01 · 기초

    LLM 추론의 기본 풍경

    본격적으로 들어가기 전에, 영상 내내 등장할 단어들의 멘탈 모델을 먼저 잡습니다. 이미 익숙하다면 빠르게 훑고 넘어가셔도 됩니다.

    LLM이 한 토큰을 만드는 한 사이클

    LLM 은 결국 “지금까지의 토큰들을 보고 다음 토큰을 예측하는” 거대한 함수입니다. 이 함수를 한 번 실행하는 일이 forward pass 이고, 이게 바로 우리가 추론(inference) 이라고 부르는 작업입니다.

    한 번의 forward pass 에서 일어나는 일을 단순화하면 이렇습니다:

    한 번의 forward pass: 토큰 시퀀스가 모든 layer 를 통과해 다음 토큰의 확률 분포를 만들어 냅니다.

    각 layer 안에서는 가중치(weights) 라는, 학습으로 결정된 거대한 행렬들이 입력과 곱해집니다. 가중치는 모든 사용자가 공유합니다 — 누가 질문하든 같은 GPT-4 가중치가 쓰입니다. 이 한 가지 사실이 모듈 3에서 모든 것의 출발점이 됩니다.

    왜 GPU 인가

    LLM 의 거의 모든 계산은 “행렬 곱하기” 입니다. 행렬 곱하기는 결과 행렬의 셀 하나하나가 서로 독립적으로 계산되기 때문에 병렬 처리에 매우 잘 맞습니다. GPU 는 정확히 이 “수많은 곱셈을 동시에 하는 일” 에 특화된 칩입니다.

    두 가지 핵심 데이터: 가중치 vs KV cache

    추론 중에 GPU 의 HBM (고대역폭 메모리) 안에 늘 머물러 있는 두 가지 큰 데이터가 있습니다:

    • 가중치 (weights): 학습으로 만들어진 고정된 숫자들. 모든 사용자가 공유합니다.
    • KV cache: 지금 진행 중인 대화의 “단기 기억”. 토큰이 추가될 때마다 누적되며, 사용자(시퀀스) 마다 따로 보관됩니다.

    이 두 데이터를 매 토큰마다 GPU 코어로 가져와서 곱셈해야 합니다. 그래서 HBM 에서 코어까지의 대역폭 이 LLM 추론의 진짜 병목이 되는 경우가 많습니다. (왜 그런지는 다음 모듈에서 자세히 다룹니다.)

    더 자세히: forward pass 안에서 정확히 일어나는 일

    각 layer 안에서 일어나는 일을 순서대로 적으면:

    1. 입력 활성화 텐서를 Q = X @ W_Q, K = X @ W_K, V = X @ W_V 로 변환 (세 번의 행렬곱)
    2. 이번 토큰의 K, V 를 KV cache 에 추가
    3. Attention 계산: softmax(Q @ K^T / √d) @ V
    4. MLP: up_projection → 비선형 함수 → down_projection
    5. 잔차 연결로 입력에 더하기 → 다음 layer 로 전달

    모든 단계가 GPU 위에서 도는 행렬곱이고, 그 행렬은 모두 HBM 에서 가져온 가중치입니다.

    02 · 핵심 직관

    두 개의 시계: 계산 시간과 메모리 시간

    영상의 모든 결론은 한 가지 질문에서 출발합니다: “GPU 는 지금 계산이 끝나길 기다리고 있는가, 아니면 데이터가 도착하길 기다리고 있는가?”

    한 토큰을 만드는 동안 GPU 가 하는 두 가지 일

    한 토큰을 만들기 위해 GPU 가 하는 일은 크게 두 가지로 나뉩니다:

    1. 계산 (compute) — 행렬곱과 덧셈. 처리 속도의 단위는 FLOP/s (초당 연산 횟수).
    2. 데이터 가져오기 (memory fetch) — HBM 에 있는 가중치와 KV cache 를 GPU 코어 쪽으로 옮기기. 단위는 GB/s.

    이 두 작업은 GPU 안에서 거의 병렬로 일어납니다 (계산하는 동안 다음 데이터를 미리 가져옴). 그래서 한 단계가 걸리는 시간은 두 작업 중 더 오래 걸리는 쪽 으로 결정됩니다.

    법칙 ① t_step = max( t_compute, t_memory )

    두 시계가 어떻게 다른가

    긴 쪽이 늘 병목. 짧은 쪽은 그 시간 동안 “놀고 있는” 셈입니다.

    실제 숫자로 직관 잡기

    Nvidia B100 의 사양을 봅시다: 계산 처리량 약 4.5 PFLOP/s (초당 4.5 × 10¹⁵ 회 연산), 메모리 대역폭 약 8 TB/s.

    이제 “HBM 에서 1 MB 짜리 가중치를 가져와 한 번 곱하기만 하는” 작업을 상상해 봅시다:

    • 1 MB 가져오는 시간: 1 ÷ (8 × 10⁶ MB/s) ≈ 125 나노초
    • 1 MB 가중치 (≈ 100만 개 숫자) 로 한 번씩 곱하기: 10⁶ ÷ 4.5 × 10¹⁵ ≈ 0.0002 나노초

    메모리가 약 60만 배 느립니다. 이런 상황에서는 GPU 코어가 데이터를 받자마자 곱셈을 끝내고 거의 매 순간 다음 데이터를 기다리게 됩니다. 자연스럽게 “이왕 가중치 한 번 가져왔으니, 그걸로 더 많은 일을 시키자” 는 발상이 등장하죠 — 그게 다음 모듈의 batch 입니다.

    한 줄 요약

    Roofline: 어떤 작업이든 “피크 계산력”과 “데이터 전송 한계” 둘 중 더 작은 쪽에 묶입니다. LLM 추론은 자주 메모리에 묶이는데, 다음 모듈에서 보겠지만, 그 병목을 푸는 도구가 batch 입니다.

    더 자세히: 왜 두 작업 중 “긴 쪽” 시간이 한 단계 시간이 되는가

    GPU 는 “double buffered” 방식으로 동작합니다. 한 데이터 청크를 계산하는 동안 다음 청크의 데이터를 미리 가져옵니다 (prefetch). 그래서 매 청크마다 계산이 5 ms 걸리고 데이터 가져오기가 8 ms 걸리는 경우, 전체 N 개 청크 처리 시간은 5 + 8 + 8 + ... + 8 = 8N + 5 ≈ 8N. 즉 긴 쪽 시간 × 청크 수 로 수렴합니다.

    이게 “max(compute, memory)” 라는 표현의 정확한 의미입니다.

    03 · 본격 시작

    Batch 의 마법 — 1000배의 경제

    Reiner 는 영상의 첫 30분을 오로지 “batch” 한 단어에 씁니다. 그만큼 모든 결론이 여기서 흘러나옵니다. 이 단어가 이 맥락에서 정확히 무엇을 뜻하는지부터 짚고 갑시다.

    이 맥락에서의 “batch” 정의

    “Batch” 는 컴퓨팅 전반에서 흔히 쓰여 의미가 모호한 단어인데, 이 영상에서의 정확한 뜻 은 이렇습니다:

    정의

    Batch = 동시에 도착한 여러 사용자(또는 시퀀스)의 forward pass 를 한 번의 GPU 실행에 묶어서 처리하는 일. “batch size 2,400” 이라면 “지금 ChatGPT 에 동시에 질문 중인 2,400명을 한 번의 forward 로 함께 처리” 한다는 의미입니다.

    왜 묶으면 싸지는가

    모듈 1에서 강조했듯이, 가중치는 모든 사용자가 공유합니다. 그러니 GPU 가 가중치를 HBM 에서 한 번 가져오기만 하면, 그 한 번의 fetch 로 “2,400명의 토큰” 을 함께 곱할 수 있습니다. 한 번의 fetch 비용을 N 명에게 나눠 부담시키는 셈인데, 이걸 amortize (분할 상환) 한다고 합니다.

    토큰 1개당 가중치 fetch 비용을 식으로 쓰면:

    법칙 ② 토큰당_가중치_비용 = (가중치_fetch_시간) / batch_size

    batch size = 1 → 한 사람이 가중치 비용을 통째로 떠안음. 토큰당 가격이 천문학적이 됩니다.
    batch size = 1,000 → 천 명이 1/1000 씩 나눠 부담. 거의 무시 가능한 수준이 됩니다.

    비용 곡선의 모양

    batch 가 작을 때는 가중치 비용이 쌍곡선 모양으로 가파르게 떨어지고, 어느 지점에 도달하면 계산 시간이 만든 floor 에 닿아 평탄해집니다. 그 너머로 더 키우면 사용자별 KV cache 부담이 누적되어 비용이 다시 끌어올려집니다.

    Reiner 가 영상에서 “batching 이 1,000배 차이를 만든다” 고 말하는 이유가 여기 있습니다: 같은 GPU, 같은 모델이라도 batch 정책에 따라 토큰당 가격이 1000배 가까이 벌어질 수 있습니다. “ChatGPT 무료 사용자 vs OpenAI API 호출” 사이의 단가 차이도 이런 batch 효율의 결과입니다.

    그럼 batch 를 무한정 키울 수 있나?

    아닙니다. KV cache 는 사용자마다 따로 보관되기 때문입니다 (모듈 8에서 자세히). batch 가 N 이면 KV 메모리도 N 배가 됩니다. 어느 순간부터는 “가중치 fetch” 가 아니라 “KV fetch” 가 새로운 병목이 되어, batch 를 더 키워도 토큰당 비용이 떨어지지 않고 오히려 올라가는 구간이 옵니다.

    그래서 “딱 좋은 batch size” 가 존재합니다. 다음 모듈에서 그 균형점이 어디인지 정확히 도출해 봅니다.

    04 · 균형점

    마법의 숫자 300

    어떤 칩 세대를 쓰든 한 숫자가 거의 변하지 않습니다. 이 숫자가 모든 batch 결정의 기준점이 됩니다.

    FLOPs / bandwidth 비율

    모든 GPU 의 데이터시트에는 두 숫자가 함께 적혀 있습니다: “초당 가능한 연산 횟수” (FLOP/s) 와 “초당 가져올 수 있는 데이터양” (byte/s). 이 둘의 비율 — “HBM 에서 1 byte 를 가져오는 시간 동안 GPU 코어가 몇 번의 곱셈을 할 수 있는가” — 가 그 하드웨어의 성격을 결정합니다.

    법칙 ③ — 마법의 숫자 FLOPs / memory_bandwidth ≈ 300 (단위 없음)

    재미있는 건 이 비율이 GPU 세대가 바뀌어도 거의 그대로 유지된다 는 사실입니다:

    • A100 (2020): ≈ 300
    • H100 (2022): ≈ 290
    • B100 (2024): ≈ 300

    이유는 단순합니다: GPU 코어와 HBM 은 같은 칩 안에서 transistor, 면적, 전력을 두고 서로 경쟁 합니다. 한쪽만 비대해지면 비용 효율이 무너지니, 결국 시장이 두 자원의 균형을 유지하도록 압력을 가하게 됩니다.

    이 숫자에서 “최적 batch size” 가 도출된다

    모듈 3의 “batch 가 너무 작으면 가중치 비용이 압도한다” 를 수식으로 다시 쓰면:

    균형 조건 (가중치 fetch 시간 ≈ 가중치 곱셈 시간) total_params / bandwidth = batch × active_params / FLOPs

    좌변은 가중치를 HBM 에서 가져오는 데 걸리는 시간 (사용자 수와 무관), 우변은 batch 명의 토큰에 대해 active params 만큼 곱셈하는 시간입니다. 양변을 정리해 모델 항과 하드웨어 항을 분리하면:

    FLOPs / bandwidth = batch × (active_params / total_params) ⇒ batch_optimal ≈ 300 × (total_params / active_params) = 300 × sparsity

    여기서 sparsity = total_params / active_params 입니다. Dense 모델의 경우 active = total 이므로 sparsity = 1, 즉 균형 batch 는 약 300. MoE 모델은 sparsity 가 클수록 batch 도 그에 비례해 커야 합니다.

    실제 모델로 계산해 보기

    DeepSeek 은 256개 expert 중 32개만 활성화합니다. 즉 sparsity = 256 / 32 = 8.

    batch_optimal ≈ 300 × 8 = 2,400 sequences

    이것이 frontier serving stack 이 보통 “수천 명 단위의 batch” 로 운영되는 이유입니다. 이 batch 효율은 GPU 가 “warm” 상태일 때만 유지되므로, 회사들은 대략 20 ms 마다 한 batch 를 출발시키는 식으로 시간표를 운영합니다 (HBM 을 한 번 다 읽어내는 데 ~15-20 ms 가 걸리기 때문).

    왜 이게 강력한가

    “최적 batch” 공식이 active params, total params, layer 수, head 수 같은 모델 디테일을 거의 타지 않고 “300 × sparsity” 한 줄로 정리됩니다. 모델이 바뀌어도, GPU 세대가 바뀌어도 이 공식이 거의 그대로 적용됩니다. Reiner 가 “공개된 sparsity 만 알면 batch 를 추정할 수 있다” 고 자신 있게 말하는 이유입니다.

    더 자세히: 왜 “가중치 fetch ≈ 가중치 곱셈” 이 최적인가

    GPU 가 가능한 한 놀지 않게 하려면, 메모리가 다음 데이터를 가져오는 동안 코어는 직전 청크를 정확히 다 곱하고 있어야 합니다. 그러려면 “가져오는 시간 = 곱하는 시간” 이 되어야 합니다. 이 지점이 바로 균형점입니다.

    이 점을 넘으면 코어가 일을 다 끝내고 다음 데이터가 도착하길 기다립니다 (compute-bound 상태). 이 점에 못 미치면 데이터가 도착해도 코어가 곱셈을 따라잡지 못합니다 (memory-bound 상태). “300 × sparsity” 가 바로 그 평형 지점입니다.

    05 · 아키텍처

    Mixture of Experts — 전문가 군단

    GPT-4 이후 frontier 모델의 표준 구조입니다. “sparsity” 라는 말이 영상 내내 등장하는 것도 모두 MoE 때문입니다.

    Dense 모델 vs MoE 모델

    “Dense” 모델 (예: 초기 GPT-3, 작은 LLaMA 모델들) 은 모든 토큰이 모든 가중치를 통과 합니다. 그래서 모델 크기가 커지면 토큰당 계산량도 그에 비례해 커집니다.

    MoE 는 한 layer 안에 작은 신경망(expert) 을 여러 개 두고, “이 토큰은 expert 7번만, 저 토큰은 expert 23번만” 식으로 가중치의 일부만 골라 사용 하는 구조입니다. 한 layer 에 expert 가 256개 있다면, 한 토큰은 그 중 32개 정도만 통과합니다.

    한 MoE layer의 흐름

    Router 가 토큰을 보고 “이 토큰은 expert 2번과 7번으로 보내자” 라고 결정합니다. 그 두 expert 만 토큰을 처리한 뒤, 결과를 합쳐 다음 layer 로 넘깁니다.

    핵심 어휘 정리

    • Router: “이 토큰을 어떤 expert 에게 보낼까?” 를 결정하는 작은 신경망. 학습 과정에서 그 라우팅 정책이 자동으로 결정됩니다.
    • Expert: 작은 MLP. 한 layer 에 보통 수십~수백 개가 들어 있습니다.
    • Active params: 한 토큰을 처리할 때 실제로 통과하는 가중치의 개수.
    • Total params: 모델의 모든 expert 가중치 합. GPU 메모리에는 이 전체를 늘 올려놓아야 합니다.
    • Sparsity = total / active. 이 값이 클수록 “모델 전체는 거대하지만 토큰당 계산은 작다” 의 효과가 강해집니다.

    왜 MoE 가 “큰 모델인데 토큰당 추론은 작다” 인가

    메모리 점유는 total params 에 비례 합니다 — 어떤 토큰이 어떤 expert 로 갈지 미리 알 수 없으므로, 모든 expert 의 가중치를 GPU 에 미리 올려놓아야 합니다. 그래서 GPU 의 HBM 용량이 중요해집니다.

    계산 비용은 active params 에 비례 합니다 — 한 토큰은 32개 expert 만 통과하기 때문입니다. 그래서 같은 품질의 dense 모델에 비해 토큰당 계산량이 훨씬 적습니다.

    이 “메모리 점유 vs 계산량의 분리” 가 바로 모듈 4의 “batch ≈ 300 × sparsity” 공식에 그대로 반영됩니다. sparsity 가 클수록 batch 도 크게 돌려야 GPU 가 놀지 않게 되는 것입니다.

    06 · 하드웨어

    GPU 랙과 두 종류의 네트워크

    “한 랙(rack) 안인가, 밖인가” 가 모든 분산 결정의 출발점입니다. 이 한 줄을 이해하면 영상 후반부의 절반은 따라갈 수 있습니다.

    데이터센터의 단위

    • GPU 한 대 (예: B100 1개, ~192 GB HBM)
    • 트레이 / 노드 (예: Hopper 시대의 8 GPU 한 트레이)
    • 랙(rack) — Blackwell 세대 기준 64-72 GPU. 서버용 캐비닛 한 칸을 가리킵니다.
    • 데이터센터 (랙 수백~수천 개, 총 수만 대의 GPU)

    두 종류의 네트워크

    같은 랙 안의 GPU끼리는 NVLink로 빠르게 통신, 랙을 넘으면 데이터센터 이더넷으로 ~8배 떨어짐.
    • Scale-up: 랙 안 의 NVLink 연결망. 기준 속도 1× 의 빠른 통신.
    • Scale-out: 랙 사이 의 데이터센터 이더넷. scale-up 대비 약 8배 느림. 옆 랙으로 데이터를 보내는 일은 그만큼 비싼 일이 됩니다.

    왜 한 랙에 GPU 를 더 못 넣는가

    한 랙의 크기를 제한하는 물리적 요인들 (아래로 갈수록 빡빡한 한계):

    • 전력 공급 (랙당 100 kW 이상)
    • 구조적 무게와 랙을 지탱하는 강철 프레임
    • 냉각 (이미 액체 냉각이 필요한 단계)
    • 케이블 밀도와 backplane (후면 연결판) 의 커넥터 핀 밀도 ← 가장 빡빡한 진짜 한계

    세대별 “한 scale-up 도메인” 크기 변화

    • Hopper (2022): 트레이 한 칸 = 8 GPU
    • Blackwell (2024): 랙 한 칸 = 64–72 GPU
    • Rubin (예정): ~500 GPU. 케이블 밀도가 약 4배 향상된 결과입니다.

    Hopper → Blackwell 의 “8 → 72” 는 사실 폼팩터를 트레이에서 랙으로 바꾼 영향이 큽니다. Blackwell → Rubin 의 “72 → 500” 이 비로소 진짜 케이블 밀도의 기술적 도약입니다. Reiner 의 표현을 빌리면, “4배 향상된 것은 케이블 쪽이지 transistor 쪽이 아니다” 라는 셈입니다.

    왜 이 한 줄이 중요한가

    “모델이 한 scale-up 도메인 안에 다 들어가는가?” — 이 한 질문이 모델 분산의 거의 모든 것을 결정합니다. 들어가면 빠른 NVLink 만으로 끝납니다. 들어가지 않으면 다음 모듈에서 다룰 pipeline parallelism 이 필요해집니다.

    07 · 분산

    Pipeline Parallelism — 레이어를 나누는 법

    한 랙에 모델이 다 들어가지 못하면 어떻게 할까요? layer 를 잘라서 여러 랙에 나눠 담는 것이 가장 자연스러운 답입니다.

    기본 아이디어

    모델이 100 개의 layer 로 이루어져 있다고 합시다. layer 를 25 개씩 묶어서 4 개의 랙에 나눠 담을 수 있습니다:

    • Rack 0: layer 0–24
    • Rack 1: layer 25–49
    • Rack 2: layer 50–74
    • Rack 3: layer 75–99

    토큰 시퀀스가 rack 0 의 layer 0–24 를 통과하면, 그 결과 (“활성화 텐서”) 를 rack 1 로 넘깁니다. rack 1 이 layer 25–49 를 처리한 뒤 rack 2 로 넘기고, 그렇게 마지막 랙까지 이어집니다 — 마치 공장 컨베이어와 같은 구조입니다. 이것이 pipeline parallelism 입니다.

    그런데 “랙 사이는 8배 느리다” 고 하지 않았나?

    맞습니다. 그래서 랙 사이로 넘기는 데이터의 양 과 랙 안에서 도는 데이터의 양 의 비율이 중요해집니다. 랙 안에서 충분히 많은 작업을 처리하고 가끔씩만 옆 랙으로 넘긴다면, 그 “가끔” 의 비싼 비용이 전체 시간 안에 잘 묻혀 보이지 않게 됩니다.

    MoE 와 결합된 경우의 부등식은 이렇게 나옵니다:

    손해 보지 않는 조건 (랙 안에서 도는 작업 시간) ≥ (랙 사이 통신 시간) ⇒ 8 × (활성 expert 수) × (한 stage 의 layer 수) ≥ 8

    좌변의 8 은 “scale-up 이 scale-out 보다 8배 빠르다” 는 보정 계수, 우변의 8 은 한 stage 가 보내야 할 활성화 텐서 양에 all-to-all 통신의 안전 마진을 더한 보수적 추정치입니다. DeepSeek (활성 expert 8) 에 stage 당 layer 8 개를 두면 좌변 = 8 × 8 × 1 = 64 ≥ 8 → 조건이 여유 있게 만족됩니다. “한 랙이 한 layer 만 담당” 하는 극단적 분배에서도 이 부등식은 충분히 성립합니다.

    깔끔한 결과: “한 랙 = 한 layer”

    이 부등식이 워낙 여유롭게 만족되기 때문에, 실제 frontier serving stack 에서는 모델의 layer 경계와 데이터센터의 랙 경계가 자연스럽게 맞아떨어집니다. 모델 아키텍처가 곧 물리적 배치를 결정하는 셈입니다. Reiner 의 표현을 빌리면 “cutting matches the model architecture” 라는 것입니다.

    하지만

    “기술적으로 가능하다” 와 “실제로 좋은 선택이다” 는 다른 이야기입니다. 모듈 9에서 Ilya 가 말한 “Pipelining is not wise” 의 진짜 의미를 자세히 파헤쳐 봅니다.

    08 · 핵심 개념

    KV Cache — 모델의 단기 기억

    영상에서 가장 자주 등장하는 단어입니다. “메모리 비용은 결국 KV cache 때문에 비싸진다.” 이 모듈을 끝내면 long context 가격, prompt caching, pipeline 의 한계가 모두 같은 그림으로 묶입니다.

    왜 “기억” 이 필요한가

    Self-attention 은 새 토큰을 만들 때마다 “이전의 모든 토큰들과 얼마나 관련 있는지” 를 계산해야 합니다. 구체적으로는, 새 토큰의 “쿼리 (Q)” 가 이전 토큰들 각각의 “키 (K)” 와 내적되어 가중치를 만들고, 그 가중치로 이전 토큰들의 “값 (V)” 들을 가중합 합니다.

    만약 새 토큰을 만들 때마다 prompt 전체를 다시 forward 해서 K, V 를 매번 새로 계산한다면, N 번째 토큰을 만드는 데 N 개 토큰을 처리해야 합니다 → 전체 비용이 O(N²) 로 폭발합니다.

    해결책: 캐시하기

    이미 처리한 토큰의 K 와 V 는 한 번 계산한 뒤 HBM 에 저장해 둡니다. 새 토큰은 자기 자신의 K, V 만 새로 계산해 캐시에 추가하고, 캐시되어 있는 이전 토큰들의 K, V 와 내적만 하면 됩니다. 이렇게 저장된 K, V 모음을 KV cache 라고 합니다.

    시퀀스가 길어질수록 KV cache 가 누적. 사용자마다 따로.

    크기 공식

    법칙 ④ bytes_per_token = 2 × layers × d_head × num_KV_heads × bytes_per_value
    • 2: K 와 V 두 텐서를 모두 저장하기 때문
    • layers: 모든 attention layer 마다 별도의 KV 가 필요 (보통 80–120)
    • d_head: attention head 하나의 차원 (보통 128)
    • num_KV_heads: layer 당 KV head 의 개수 (보통 1–8, GQA / MQA 같은 기법으로 줄임)
    • bytes_per_value: 한 값을 저장하는 바이트 수 (FP16/BF16 면 2)

    현재 frontier 모델 기준으로는 대략 토큰당 ~2 KB 가 일반적인 수치입니다. 이 숫자가 모듈 11에서 Gemini 가격을 거꾸로 푸는 출발점이 됩니다.

    왜 이게 모든 것을 결정하는가

    batch 가 N 명이면 KV cache 도 N 배가 됩니다. 한 사용자가 400 MB 면 1,000 명 동시 사용 = 400 GB, 이미 한 GPU 의 HBM (~192 GB) 한도를 한참 넘어섭니다. 그래서 KV cache 는 분산이 필요한데, 다음 모듈에서 보겠지만 그 분산이 pipeline parallelism 으로는 풀리지 않습니다.

    더 자세히: 왜 K, V 만 캐시하고 Q 는 캐시하지 않는가

    새 토큰을 만들 때 attention 은 “이번 토큰의 쿼리 (Q)” 가 “이전 모든 토큰의 키 (K)” 와 내적합니다. 쿼리는 오직 이번 토큰만의 것이고 한 번 쓰고 나면 다시 쓸 일이 없으므로, 캐시할 필요가 없습니다. 반면에 K 와 V 는 미래의 모든 토큰이 계속 참조해야 할 “이전 정보” 이므로 저장해 두는 가치가 있는 것입니다.

    09 · 미묘함

    “Pipelining is not wise” 의 진짜 의미

    Ilya Sutskever 가 남긴 유명한 한 마디입니다. Reiner 는 이 말이 왜 옳은지를 수식으로 풀어서 보여줍니다.

    학습에서의 명백한 문제: pipeline bubble

    학습에서는 forward pass 다음에 반드시 backward pass 가 따라옵니다. pipeline 단계가 4개라고 합시다:

    • stage 0 이 “micro-batch 1” 의 forward 를 끝내고 stage 1 로 넘깁니다.
    • stage 0 이 “micro-batch 1” 의 backward 를 받기 시작하려면, 그 micro-batch 가 stage 3 까지 forward 를 마치고 backward 가 거꾸로 돌아와야 합니다.
    • 그 사이에 stage 0 은 잠깐 할 일이 없어집니다 — 이 idle 구간이 “bubble” 입니다.

    micro-batch 를 더 잘게 쪼개거나 영리한 스케줄링 (zero-bubble, one-forward-one-backward 등) 으로 bubble 을 줄일 수는 있지만, 완전히 없애기는 어렵습니다. 이 idle 시간이 바로 학습에서 pipeline 이 비싸지는 큰 이유 중 하나입니다.

    그럼 추론에서는 어떨까?

    추론에는 backward 가 없으니 bubble 도 없습니다. 그렇다면 pipeline 이 깨끗하게 작동할까요? Reiner 는 다음 네 가지 차원에서 따져봅니다:

    차원 Pipeline 효과
    Latency (한 토큰 시간) 변화 없음. 어차피 모든 layer를 순서대로 통과 (오히려 hop 지연 추가).
    Compute (총 FLOPs) 변화 없음.
    Weight 메모리 용량 ↓ 1/P 만큼 줄어듦. (P단계로 나눴으니)
    KV cache 메모리/대역폭 변화 없음 ← 결정타.

    왜 KV cache 는 pipeline 의 효과를 받지 못하는가

    pipeline 을 P 단계로 나누면 GPU 수가 P 배가 됩니다. “가중치 메모리 용량은 1/P 로 줄어든다” 는 분명한 결과입니다. 그런데 KV cache 는 어떨까요?

    핵심은 “늘어난 P 배의 GPU 를 놀지 않게 하려면 batch 도 P 배로 늘려야 한다” 는 점입니다. 그렇지 않으면 추가된 GPU 들이 일감 없이 놀게 되어 분산의 의미가 사라지기 때문입니다. 그래서 처리 중인 시퀀스 수도 자연스럽게 B 에서 B × P 로 늘어납니다:

    GPU 한 대당 KV cache 양 KV_per_GPU = (B × P × context × bytes_per_token) / (E × P) = B × context × bytes_per_token / E

    여기서 E 는 expert parallelism 계수입니다. 분자에 P 가 들어가고 분모에도 P 가 들어가서 두 P 가 약분되어 사라집니다. 분명히 GPU 를 더 많이 펼쳤는데, GPU 한 대가 떠받쳐야 하는 KV cache 양은 변함없이 그대로이고, 메모리 대역폭 압박도 풀리지 않습니다.

    결정타

    Pipeline 은 “가중치 메모리 용량” 만 줄여 줍니다. 하지만 frontier serving 의 진짜 압박은 가중치 용량이 아니라 KV cache 의 대역폭 입니다. 그 압박은 풀어주지 못하면서 랙 간 hop 지연만 추가하니, 결과적으로 “not wise” 한 선택이 되는 것입니다.

    그래서 frontier lab 의 선택은

    가능하면 한 scale-up 도메인 안에 모델을 모두 담고, pipeline 은 최소화하는 것입니다. Nvidia 가 한 랙당 GPU 수를 끌어올리느라 (Hopper 8 → Blackwell 72 → Rubin 500) 케이블 밀도 같은 물리적 한계와 씨름하는 것도 결국 같은 이유 — 한 scale-up 안에 더 큰 모델을 담기 위해서입니다.

    더 자세히: 그럼 pipeline 은 언제 쓰는가

    모델의 가중치 총량이 한 scale-up 도메인의 HBM 합을 넘어설 때는 어쩔 수 없이 pipeline 을 씁니다. 예: 5 T 파라미터 모델 + 한 랙의 HBM 합 ≈ 13 TB 인 경우, 약 4 랙 정도의 pipeline 이 필요합니다.

    그러나 한 도메인 안에 들어갈 수만 있다면, pipeline 의 “KV 압박은 못 풀고 hop 지연만 추가” 라는 비용을 피하는 것이 압도적으로 유리합니다.

    10 · 학습 경제학

    Chinchilla 와 100배 오버트레이닝

    “파라미터 1개당 토큰 20개” 라는 옛 권고는 frontier 모델들에서 사실상 무시되고 있습니다. 이유는 단순합니다: 학습 비용보다 평생 추론 비용이 훨씬 큽니다.

    Chinchilla 란?

    2022년 DeepMind 논문이 제시한 “compute-optimal” 배합입니다: 파라미터 N 개의 모델은 약 20 × N 토큰으로 학습할 때 같은 학습 FLOPs 예산에서 최고 성능이 나옵니다.

    • 70 B 모델 → 약 1.4 T 토큰이 compute-optimal
    • 100 B 모델 → 약 2 T 토큰이 compute-optimal

    그런데 실제 frontier 모델은

    추정에 따르면 GPT-5 나 Claude 4 같은 모델은 100 T 단위의 토큰으로 학습되었습니다. 100 B 파라미터 모델의 Chinchilla 권고치가 2 T 인데, 실제 학습량은 그보다 50–100배 더 많은 셈입니다. 이걸 두고 “100배 over-trained” 라고 부릅니다.

    이것을 어떻게 합리화할 수 있을까요? Reiner 의 답은 “평생 추론 비용까지 고려한 최적화” 입니다.

    비용 균등화 휴리스틱

    \(f(x) = 1/x + x\) 라는 함수는 \(x = 1\) 에서 최소가 됩니다. 즉 두 항이 같아질 때가 최소입니다. 이와 비슷하게, “total_cost = train_cost + infer_cost” 처럼 한쪽을 늘리면 다른 쪽이 줄어드는 trade-off 구조에서는 두 비용이 비슷해지는 지점 근처가 최적이 되는 경우가 많습니다.

    그래서 frontier lab 은 다음 세 비용이 비슷한 규모가 되도록 균형을 맞춥니다:

    3-way 균등화 Pretraining 비용 ≈ RL 비용 ≈ 평생 Inference 비용

    각 비용을 FLOPs 로 풀어 쓰면:

    • Pretraining: 6 × N_active × D_pre (forward 3 + backward 3 FLOPs/param/token)
    • RL: α × N_active × D_RL (α 는 2 ~ 6)
    • Inference: 2 × N_active × D_inf (forward 만, backward 없음)

    α 가 2 와 6 사이인 이유: RL 의 토큰은 대부분 “모델이 직접 생성한 후보 응답들” 이라 상당량이 generation (forward 만) 이고, 그 중 일부에만 backward 가 따라붙기 때문입니다. 사실상 “추론에 가까운 학습” 인 셈입니다.

    실제 숫자로 100배가 나오는 과정

    2025년 frontier 모델 추정치로 계산해 봅시다:

    • 한 모델의 평생 추론 토큰: D_inf ≈ 200 T
    • Active params: N_active ≈ 100 B
    • Inference FLOPs = 2 × 100 × 10⁹ × 200 × 10¹² = 4 × 10²⁵ FLOPs
    • 같은 규모로 pretraining 비용을 맞추려면 D_pre 가 (2/6) × 200T ≈ 67 T 토큰. 실제 Pope 의 추정 (~150 T) 과 같은 자릿수입니다.
    • Chinchilla 권고는 20 × 100 B = 2 T 토큰
    결과 D_실제 / D_chinchilla ≈ 100T / 2T = 50–100 배 over-trained
    Reiner 의 한 줄

    “각 모델은 자신이 받는 모든 입력에 대한 응답으로, 인류가 지금까지 쓴 모든 글의 합과 맞먹는 양의 토큰을 생성한다.” — 이것이 100배 over-training 의 정성적 의미입니다.

    더 자세히: 왜 비용 균등화가 최적인가

    비용을 \(C(N, D) = a · N + b / N\) 처럼 한쪽 항은 N 에 비례하고 다른 항은 1/N 에 비례하는 형태로 쓸 수 있을 때, N 에 대해 미분해서 0 으로 놓으면 두 항이 같아지는 지점에서 합이 최소가 됩니다.

    학습 / 추론 trade-off 도 비슷합니다: 모델을 작게 (N ↓) 만들면 평생 추론 비용은 줄지만 같은 품질을 유지하려면 학습 토큰 D 가 더 필요해 학습 비용이 늘어납니다. 두 비용이 비슷한 자릿수가 될 때 총 비용이 최소에 가까워집니다.

    11 · 가격 리버스 엔지니어링

    Long Context 의 경제학

    “Gemini 가 200K 컨텍스트에서 가격을 50% 올린다” — 이 한 줄에 담긴 정보가 놀랍도록 풍부합니다. 이 모듈에서는 그 한 줄로부터 모델의 내부 구조까지 거꾸로 풀어내는 과정을 보여줍니다.

    가격 곡선의 형태

    컨텍스트가 짧을 때는 가격이 평탄하다가, 어느 지점에서 KV fetch 비용이 가중치 fetch 비용을 추월하면서 비용이 가파르게 올라갑니다. Gemini 는 바로 그 “꺾이는 지점” 에 가격 분기를 둡니다.

    Gemini 2.x 의 가격 정책은 다음과 같습니다:

    • 컨텍스트 < 200K: 1× 가격
    • 컨텍스트 ≥ 200K: 1.5× 가격

    이건 단순한 마케팅이 아니라 그 지점이 “메모리 fetch 시간 = 계산 시간” 의 inflection point 라는 운영적 신호입니다.

    거꾸로 푸는 수학

    균형점에서는 KV fetch 시간이 계산 시간과 같아집니다:

    균형 조건 (B × L × bytes_per_token) / bandwidth = (B × N_active) / FLOPs

    양변에서 B 를 약분하고 정리하면:

    L × bytes_per_token = N_active × (bandwidth / FLOPs) = N_active / 300 ← 모듈 4 의 마법의 숫자!

    L = 200K, N_active = 100B 를 대입해 bytes_per_token 에 대해 풀면:

    bytes_per_token = 100 × 10⁹ / (200,000 × 300) ≈ 1,667 bytes ≈ 2 KB

    이 값이 모듈 8 에서 본 KV cache 공식 “2 × layers × d_head × num_KV_heads × bytes_per_value” 의 결과와 정확히 일치합니다. Gemini 의 가격 한 줄로부터 모델의 head 구조까지 거꾸로 풀려 나오는 셈입니다.

    또 다른 신호 ① — Prefill vs Decode 가격차

    대부분의 API 가격에서 출력 토큰이 입력 토큰보다 5–10배 비쌉니다. 왜 그럴까요?

    • Prefill: 사용자 prompt N 개의 토큰을 한 번에 처리합니다. 가중치를 한 번 fetch 한 비용으로 N 개의 토큰을 모두 처리할 수 있어, 토큰당 가중치 비용이 1/N 로 줄어듭니다.
    • Decode: 한 번에 한 토큰씩 생성합니다. 매 토큰마다 가중치를 다시 fetch 해야 하므로 토큰당 가중치 비용이 1 로 통째 듭니다.

    가격비가 5–10배라는 것은 “가중치 fetch 비용이 전체 비용의 5–10배 비중을 차지한다” 는 뜻 — 즉 frontier 추론이 심하게 메모리에 묶여 있다 는 또 하나의 증거입니다.

    또 다른 신호 ② — Cache Hit vs Cache Miss

    많은 API 가 prompt caching 을 제공합니다. 같은 prefix 를 반복해서 보내면 두 번째 요청부터는 약 10배 싸지는 식입니다.

    • Cache miss = prefill 을 다시 수행 (전체 forward pass 한 번)
    • Cache hit = HBM 에서 KV 만 읽어오기
    • 10배 차이 → HBM 접근 비용이 한 번의 forward 비용의 약 1/10
    한 점에서 풀려 나온 것들

    “Gemini 가 200K 에서 가격을 50% 올림” 한 줄로부터 — KV cache 크기, 모델의 head 구조, “심하게 메모리에 묶여 있다” 는 운영 사실, prompt caching 의 경제적 합리성까지 줄줄이 풀려 나옵니다. 이것이 Reiner 가 보여주는 “공개된 API 가격으로 frontier lab 을 reverse engineering” 의 정수입니다.

    왜 컨텍스트가 200K 에서 멈춰 있는가

    현재 frontier 모델의 “실효 컨텍스트” 길이는 1–2년째 100–200K 근방에서 정체 상태입니다. 이유는 다름 아닌 메모리 대역폭입니다. 계산 비용은 컨텍스트 길이에 거의 선형으로만 늘어나지만, 메모리 fetch 비용이 200K 즈음에서 “batch 를 줄여야만 처리 가능” 한 영역으로 들어가면서 토큰당 단가가 가파르게 올라가기 때문입니다.

    Sparse attention 같은 기법으로 실효 컨텍스트를 더 늘릴 여지는 있지만 무한정은 아니고, “HBM 의 용량과 대역폭은 그대로” 라는 물리적 한계가 결정적입니다. 만약 AGI 의 in-context learning 이 100 M 토큰 정도의 컨텍스트를 요구할 것 이라는 가설이 맞다면, 현재 ML 인프라에서는 매우 큰 돌파구가 필요하다는 의미가 됩니다.

    12 · 종합

    한 장의 그림

    지금까지 다룬 모든 개념이 어떻게 한 의사결정 안에서 서로 맞물리는지 살펴봅시다. 영상의 거의 모든 내용이 이 한 장으로 압축됩니다.

    frontier serving 의 “4 가지 숫자 + 1 가지 휴리스틱”

    영상의 거의 모든 결론은 다음에서 흘러나옵니다:

    숫자 출처 결정하는 것
    FLOPs/bandwidth ≈ 300 GPU 데이터시트 최적 batch size, 가격 inflection
    Sparsity 모델 카드 (active/total) batch ≈ 300 × sparsity
    Layer 수, head 차원 모델 카드 bytes/token, 가능한 컨텍스트 길이
    Total params 모델 카드 한 랙으로 가능한지, pipeline 필요 여부
    비용 균등화 휴리스틱 경제적 추론 100배 over-training, RL/inference 분배

    의사결정의 흐름

    1. 모델이 한 scale-up 도메인에 다 들어가는가? — 들어가면 expert parallelism + all-to-all 통신만으로 충분합니다. 들어가지 않을 때만 pipeline 을 사용합니다.
    2. 최적 batch 는? — 300 × sparsity. KV 메모리 한도 안에서 그만큼 사용자를 모읍니다.
    3. 토큰당 가격은? — 토큰당 가중치 fetch 시간 / batch + 토큰당 KV fetch 시간. long context 에서는 후자가 빠르게 압도하게 되어 가격에 inflection 이 생깁니다.
    4. Pipeline 은 가급적 쓰지 않습니다 — Latency 도 줄지 않고, KV 압박도 풀어주지 못하며, 오직 가중치 메모리 용량만 줄여줄 뿐이기 때문입니다. 한 도메인에 가중치가 다 들어가지 않을 때만 어쩔 수 없이 사용합니다.
    5. 학습은 Chinchilla 의 50–100배로 over-train 합니다 — 평생 추론 비용이 학습 비용보다 훨씬 크기 때문에, “작은 모델 + 더 길게 학습” 이 총 비용을 줄여 줍니다.
