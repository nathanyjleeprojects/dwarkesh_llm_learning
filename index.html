    01 · 기초

    LLM 추론의 기본 풍경

    본격 들어가기 전에, 영상 내내 등장할 단어들의 mental model을 잡습니다. 이미 익숙하다면 빠르게 훑고 넘어가셔도 됩니다.

    LLM이 한 토큰을 만드는 한 사이클

    LLM은 결국 “지금까지의 토큰들을 보고 다음 토큰을 예측하는” 거대한 함수입니다. 이 함수를 한 번 실행하는 게 forward pass이고, 이게 바로 우리가 “추론(inference)”이라고 부르는 일입니다.

    한 forward에서 일어나는 일을 단순화하면 이렇습니다:

    한 forward pass: 토큰 시퀀스가 모든 layer를 통과해 다음 토큰의 확률 분포를 만든다.

    각 layer 안에서는 가중치(weights)라는, 학습으로 결정된 거대한 행렬들이 입력과 곱해집니다. 가중치는 모든 사용자가 공유합니다 — 누가 질문하든 같은 GPT-4 가중치가 쓰입니다. 이 사실이 곧 모듈 3에서 polynomial하게 중요해질 겁니다.

    왜 GPU 인가

    LLM의 거의 모든 계산은 “행렬 곱하기”입니다. 행렬 곱하기는 셀 하나하나가 다른 셀과 독립적으로 계산되니까, 병렬 처리가 천국처럼 잘 됩니다. GPU는 정확히 그걸 잘하는 칩입니다.

    두 가지 핵심 데이터: 가중치 vs KV cache

    추론 시 GPU의 HBM(고대역폭 메모리) 안에 머무는 두 가지 큰 데이터가 있습니다:

    • 가중치 (weights): 학습으로 만들어진 고정된 숫자들. 사용자 모두가 공유.
    • KV cache: 지금 진행 중인 대화의 “단기 기억”. 토큰마다 누적되며, 사용자(시퀀스)마다 따로.

    이 둘이 매 토큰마다 코어로 끌려 들어와야 합니다. 그래서 HBM에서 코어까지의 대역폭이 LLM 추론의 진짜 병목이 자주 됩니다. (왜 그런지는 다음 모듈에서.)

    더 자세히: “forward pass” 안에서 정확히 일어나는 일

    각 layer 안에서:

    1. 입력 활성화 텐서를 Q = X @ W_Q, K = X @ W_K, V = X @ W_V 로 변환 (3개의 행렬곱)
    2. K, V를 KV cache 에 append
    3. Attention: softmax(Q @ K^T / sqrt(d)) @ V
    4. MLP: up_projection → 비선형 → down_projection
    5. 잔차 연결로 입력에 더함 → 다음 layer로

    모든 단계가 GPU에서 행렬곱이고, 전부 가중치를 HBM에서 끌어옵니다.

    02 · 핵심 직관

    두 개의 시계: 계산 시간과 메모리 시간

    영상의 모든 결론은 한 가지 질문에서 출발합니다: “GPU는 지금 계산을 기다리고 있나, 아니면 데이터를 기다리고 있나?”

    한 토큰에 두 가지 활동

    한 토큰을 만들 때 GPU는 두 가지 일을 합니다:

    1. 계산하기 (compute) — 행렬곱과 덧셈. 단위는 FLOPs.
    2. 데이터 가져오기 (memory fetch) — HBM에서 가중치, KV cache를 코어로. 단위는 GB/s.

    이 둘은 GPU 안에서 거의 병렬로 일어납니다 (계산하면서 다음 데이터를 미리 가져옴). 그래서 한 단계의 시간 = 둘 중 더 긴 쪽입니다.

    법칙 ① t_step = max( t_compute, t_memory )

    두 시계가 어떻게 다른가

    긴 쪽이 늘 병목. 짧은 쪽은 그 시간 동안 “놀고 있는” 셈.

    실제 숫자로 직관 잡기

    Nvidia B100의 사양을 봅시다: 약 4.5 PFLOP/s (4.5 × 10¹⁵ 연산/초), 메모리 대역폭 약 8 TB/s.

    “1MB의 가중치를 가져와서 한 번 곱한다” 라는 작업을 생각해봅시다:

    • 1MB 가져오는 시간: 1 / (8 × 10⁶ MB/s) = 125 나노초
    • 1MB 가중치(≈1M개)로 1번씩 곱하기: 1M / 4.5e15 = 0.0002 나노초

    메모리가 60만 배 느립니다. 이런 상황에서는 GPU 코어가 거의 매 순간 데이터를 기다립니다. “가중치를 한 번 가져왔으면, 그걸로 더 많은 일을 시키자” 는 발상이 자연스러워지죠.

    한 줄 요약

    Roofline: 어떤 작업이든 “피크 계산력”과 “데이터 전송 한계” 둘 중 더 작은 쪽에 묶입니다. LLM 추론은 자주 메모리에 묶이고, 다음 모듈에서 보겠지만, 그 묶임을 푸는 게 batch입니다.

    더 자세히: 왜 “둘 중 더 긴 쪽” 인가

    GPU는 “double buffered” 구조로 동작합니다. 한 데이터 청크를 계산하는 동안 다음 청크의 데이터를 미리 가져옵니다 (prefetch). 그래서 만약 매 청크에서 계산이 5ms 걸리고 데이터 가져오기가 8ms 걸리면, 전체 N개 청크 시간은 5+8+8+8+...+8 = 8N+5 ≈ 8N. 즉 긴 쪽 시간 × 청크 수로 수렴합니다.

    이게 “max(compute, memory)” 의 정확한 의미입니다.

    03 · 본격 시작

    Batch 의 마법 — 1000배의 경제

    Reiner는 영상의 첫 30분을 “batch” 하나에 씁니다. 그만큼 모든 게 여기서 나옵니다. 이 단어가 이 맥락에서 정확히 무엇인지부터 짚고 갑시다.

    이 맥락에서 “batch” 란?

    “Batch”는 컴퓨팅에서 굉장히 흔한 단어라 의미가 모호한데, 이 영상에서의 정확한 뜻은 이렇습니다:

    정의

    Batch = 동시에 도착한 여러 사용자(또는 시퀀스)의 forward pass를 한 번에 묶어서 처리. “Batch size 2,400” 이라면 “지금 ChatGPT에 동시에 질문 중인 2,400명을 한 forward로 같이 처리” 라는 뜻.

    왜 묶으면 싸지나

    모듈 1에서 말했듯이, 가중치는 모든 사용자가 공유합니다. 그러니 GPU가 가중치를 HBM에서 한 번 끌어오면, 그 한 번의 fetch에 “2,400명의 토큰”을 같이 곱할 수 있습니다. 한 번의 fetch 비용을 N명에게 amortize 하는 거죠.

    토큰 1개당 가중치 fetch 비용을 식으로 쓰면:

    법칙 ② 가중치_비용_per_토큰 = (가중치_fetch_시간) / batch_size

    Batch size = 1 → 한 사람이 가중치 비용을 통째로 부담. 토큰당 가격이 천문학적.
    Batch size = 1,000 → 천 명이 1/1000씩 나눠 부담. 거의 무시 가능.

    비용 곡선의 모양

    Batch가 작을 땐 가중치 비용이 쌍곡선으로 떨어지고, 어느 시점부터는 계산 floor에 부딪힘. 그 너머로 더 키우면 KV cache 메모리 부담이 다시 비용을 들어올림.

    Reiner가 영상에서 “batching이 1,000배 차이를 만든다” 고 말하는 이유: 같은 GPU, 같은 모델인데 batch 정책이 다르면 토큰당 가격이 1000배 차이가 납니다. 그게 “ChatGPT 무료 vs API 호출당 $0.001” 같은 가격차의 본질입니다.

    그럼 batch를 무한정 키울 수 있나?

    아니요. KV cache는 사용자마다 따로입니다 (모듈 8에서 자세히). Batch가 N이면 KV 메모리/대역폭도 N배. 어느 순간 “가중치 fetch”가 아니라 “KV fetch”가 병목이 되면서, 더 키워도 토큰당 비용이 안 떨어지고 오히려 올라가는 영역이 옵니다.

    그래서 “딱 좋은 batch size” 가 존재합니다. 다음 모듈에서 그 “딱 좋은” 지점이 어디인지 봅니다.

    04 · 균형점

    마법의 숫자 300

    어떤 칩 세대를 쓰든 한 숫자가 거의 안 변합니다. 이게 모든 batch 결정의 기준점입니다.

    FLOPs / bandwidth 의 비율

    모든 GPU의 데이터시트에는 두 숫자가 있습니다: “초당 가능한 연산 횟수” (FLOP/s) 와 “초당 가져올 수 있는 데이터양” (B/s). 이 둘의 비율 — “1 byte를 가져오는 시간 동안 몇 번 곱셈을 할 수 있나” — 가 하드웨어의 성격을 결정합니다.

    법칙 ③ — 마법의 숫자 FLOPs / memory_bandwidth ≈ 300 (단위 없음)

    재미있는 건 이 숫자가 GPU 세대가 바뀌어도 거의 안 변한다는 사실입니다:

    • A100 (2020): ≈ 300
    • H100 (2022): ≈ 290
    • B100 (2024): ≈ 300

    이유는 단순합니다: GPU 코어와 HBM은 같은 칩의 transistor와 면적과 전력을 두고 경쟁합니다. 한 쪽만 비대해지면 비용 효율이 무너지니까, 시장이 자연스럽게 비율을 유지하게 만듭니다.

    이 숫자에서 “최적 batch size” 가 떨어진다

    모듈 3의 “batch가 너무 작으면 가중치 비용이 압도”를 수식으로 쓰면:

    균형 조건 (가중치 fetch ≈ 가중치 곱셈) total_params / bandwidth = batch × active_params / FLOPs

    양변에 약간만 정리하면 (모델 항과 하드웨어 항을 분리):

    FLOPs / bandwidth = batch × (active_params / total_params) ⇒ batch_optimal ≈ 300 × (total_params / active_params) = 300 × sparsity

    실제 모델로 계산해보기

    DeepSeek은 256개 expert 중 32개를 활성화. 즉 sparsity = 256/32 = 8.

    batch_optimal ≈ 300 × 8 = 2,400 sequences

    이게 frontier serving stack이 “수천명” batch에서 도는 이유입니다. 그리고 이 batch가 “warm” 한 동안만 가격이 좋습니다 — 그래서 회사들은 “20 ms 마다 한 batch 출발” 같은 시간표를 운영합니다 (HBM 한 번 비우는 데 ~15-20ms).

    왜 이게 강력한가

    “최적 batch” 가 active params, total params, layer 수, head 수 같은 모델 디테일을 거의 안 타고 “300 × sparsity” 한 줄로 끝납니다. 모델이 바뀌어도, GPU가 바뀌어도, 이 공식 하나면 됨. Reiner가 영상에서 “공개된 sparsity만 알면 batch를 안다” 라고 자신 있게 말하는 이유입니다.

    더 자세히: 왜 “가중치 fetch ≈ 가중치 곱셈” 이 최적인가

    최대한 GPU가 안 놀게 하려면, 메모리가 데이터를 가져오는 동안 코어가 정확히 “이전 청크” 를 다 곱하고 있어야 합니다. 그러려면 “가져오는 시간 = 곱하는 시간”. 그게 균형점입니다.

    이 점을 넘으면 코어가 일을 다 끝내고 다음 데이터를 기다리게 됩니다 (compute-bound). 이 점을 못 넘으면 데이터가 와도 코어가 못 따라잡습니다 (memory-bound). “300 × sparsity” 가 바로 그 평형 위치입니다.

    05 · 아키텍처

    Mixture of Experts — 전문가 군단

    GPT-4부터 frontier 모델의 표준. “sparsity” 라는 말이 영상 내내 등장하는 건 다 MoE 때문입니다.

    Dense vs MoE

    “Dense” 모델 (예: 옛날 GPT-3, LLaMA의 작은 버전들)은 모든 토큰이 모든 가중치를 통과합니다. 모델 크기가 커지면 토큰당 계산도 비례해 커지죠.

    MoE는 그 안에서 “이 토큰은 expert 7만 활성화, 저 토큰은 expert 23만” 처럼 가중치의 일부만 골라 쓰는 구조입니다. 한 layer에 expert가 256개 있으면, 한 토큰은 그 중 32개 정도만 통과하는 식.

    한 MoE layer의 흐름

    Router가 토큰을 보고 “expert 2와 7로” 결정. 그 두 expert만 토큰을 처리. 결과를 합쳐 다음 layer로.

    핵심 어휘 정리

    • Router: “이 토큰은 어떤 expert로?” 를 결정하는 작은 신경망. 학습으로 그 정책이 자동 결정.
    • Expert: 작은 MLP. 한 layer에 보통 수십~수백 개.
    • Active params: 한 토큰을 처리할 때 실제로 통과하는 가중치 수.
    • Total params: 모델의 모든 expert 가중치 합. 메모리에는 다 들고 있어야.
    • Sparsity = total / active. 클수록 “큰 모델인데 토큰당 작게” 가 강해짐.

    왜 MoE 가 “큰 모델인데 추론은 작다” 인가

    메모리 점유는 Total에 비례합니다 — 어떤 토큰이 어떤 expert로 갈지 미리 모르니까, 모든 expert를 GPU에 올려놓아야 함. 그래서 GPU 메모리가 큰 게 중요.

    계산 비용은 Active에 비례합니다 — 한 토큰은 32개 expert만 통과하니까. 그래서 같은 quality 의 dense 모델보다 토큰당 계산이 적음.

    이 “메모리 vs 계산의 분리” 가 모듈 4의 “batch = 300 × sparsity” 공식에 직접 들어옵니다. Sparsity가 클수록 batch도 크게 돌려야 GPU가 안 놂.

    06 · 하드웨어

    GPU 랙과 두 종류의 네트워크

    “랙(rack) 안인가 밖인가” 가 모든 분산 결정의 출발점입니다. 이 한 줄을 이해하면 영상 후반의 절반을 따라갈 수 있습니다.

    데이터센터의 단위들

    • GPU 한 대 (예: B100 1개, ~192 GB HBM)
    • 트레이/노드 (예: Hopper 시대 8 GPU)
    • 랙(rack) — Blackwell 시대 64-72 GPU. 서버용 캐비닛 한 칸.
    • 데이터센터 (수만 대 GPU)

    두 종류의 네트워크

    같은 랙 안의 GPU끼리는 NVLink로 빠르게 통신, 랙을 넘으면 데이터센터 이더넷으로 ~8배 떨어짐.
    • Scale-up: 랙 안의 NVLink. “기본 1배” 의 빠른 통신.
    • Scale-out: 랙 사이 의 데이터센터 이더넷. ~8배 느림. 만나러 가는 데 비용이 큼.

    왜 한 랙에 GPU를 더 못 넣나

    한 랙 크기를 결정하는 물리적 한계들 (가장 빡빡한 게 마지막):

    • 전력 공급 (랙당 100kW+)
    • 구조적 무게 / 내장 강철
    • 냉각 (액체 냉각 필요)
    • 케이블 밀도와 backplane 연결자 핀 밀도 ← 진짜 한계

    세대별 “한 scale-up 도메인” 크기 변화

    • Hopper (2022): 트레이 한 칸 = 8 GPU
    • Blackwell (2024): 랙 한 칸 = 64-72 GPU
    • Rubin (예정): ~500 GPU. 케이블 밀도 4배 향상의 결과

    Hopper → Blackwell의 “8 → 72” 는 사실 폼팩터(트레이 → 랙)을 바꾼 게 큽니다. Blackwell → Rubin의 “72 → 500” 이 진짜 케이블 밀도의 기술적 진보. “4배 한 것은 케이블이지 transistor가 아니다”.

    왜 이 한 줄이 중요한가

    “모델이 한 scale-up 도메인 안에 다 들어가는가?” — 이 질문이 모델 분산의 모든 것을 결정합니다. 들어가면 빠른 NVLink만 쓰면 됨. 안 들어가면 다음 모듈(pipeline)이 필요.

    07 · 분산

    Pipeline Parallelism — 레이어를 나누는 법

    한 랙에 모델이 다 안 들어가면? Layer를 잘라 여러 랙에 나눠 담는 게 가장 자연스러운 답입니다.

    기본 아이디어

    모델이 100 layer라면:

    • Rack 0: layer 0–24
    • Rack 1: layer 25–49
    • Rack 2: layer 50–74
    • Rack 3: layer 75–99

    토큰 시퀀스가 rack 0의 layer 0–24를 통과하면, 그 결과(“활성화 텐서”)를 rack 1로 보냄. Rack 1이 layer 25–49 처리, 다음 rack으로 넘김. 공장 컨베이어처럼. 이게 pipeline parallelism입니다.

    그런데 “랙 사이는 8배 느림” 이라며?

    맞습니다. 그래서 랙 사이로 넘기는 데이터양 vs 랙 안에서 도는 데이터양의 비율이 중요해집니다. 랙 안에서 충분히 많이 일하고 가끔만 옆 랙으로 넘긴다면, 그 “가끔” 의 비싼 비용이 잘 가려집니다.

    MoE와 결합한 경우 부등식이 이렇게 나옵니다:

    손해 보지 않는 조건 8 × (활성_expert 수) × (한 stage의 layer 수) ≥ 8

    좌변의 “8” 은 “scale-out이 1/8 배 느림” 보정. 우변의 “8” 은 “all-to-all” 통신의 보수적 잡설들. DeepSeek (활성 expert 8) × 8 layer per stage = 64 → 64 ≥ 8 → 통과. “한 랙이 한 layer” 같은 분배도 충분히 만족합니다.

    예쁜 결과: “한 랙 = 한 layer”

    이 부등식이 워낙 여유롭게 만족돼서, 실제 frontier serving stack에서는 레이어 경계와 랙 경계가 자연스럽게 일치합니다. 모델 아키텍처가 곧 데이터센터 배치를 결정하는 것이죠. Reiner의 표현을 빌리면 “cutting matches the model architecture”.

    하지만

    “이게 가능하다” 와 “이게 좋은 선택이다” 는 다른 이야기. 모듈 9에서 “Pipelining is not wise” 의 진짜 의미를 파헤칩니다.

    08 · 핵심 개념

    KV Cache — 모델의 단기 기억

    영상에서 가장 자주 등장하는 단어. “메모리는 결국 KV cache 때문에 비싸진다”. 이 모듈을 끝내면 long context 가격, prompt caching, pipeline 한계가 모두 같은 그림으로 묶입니다.

    왜 “기억” 이 필요한가

    Self-attention은 새 토큰을 만들 때 “이전 모든 토큰들과 얼마나 관련 있나” 를 계산해야 합니다. 구체적으로는 새 토큰의 “쿼리(Q)” 가 이전 토큰들의 “키(K)” 와 내적되고, 그 가중치로 “값(V)” 들을 합칩니다.

    만약 매 토큰마다 prompt 전체를 다시 forward 해서 K, V 를 다시 계산하면, N번째 토큰을 만드는 데 N개 토큰을 처리해야 → 전체 비용이 O(N²)로 폭발합니다.

    해결책: 캐시

    이미 본 토큰의 K와 V는 한 번 계산한 뒤 HBM에 저장합니다. 새 토큰은 자기 K, V만 계산해서 캐시에 추가하고, 캐시된 것들과 내적만 하면 됩니다. 이게 KV cache.

    시퀀스가 길어질수록 KV cache 가 누적. 사용자마다 따로.

    크기 공식

    법칙 ④ bytes_per_token = 2 × layers × d_head × num_KV_heads
    • 2: K와 V 둘 다 저장
    • layers: 모델의 모든 attention layer
    • d_head × num_KV_heads: 각 layer의 head 차원 합

    현재 frontier 모델 기준으로 대략 ~2 KB / 토큰 가 “상식적” 인 값. 이 숫자가 모듈 11에서 Gemini 가격을 거꾸로 푸는 데 등장합니다.

    왜 이게 모든 걸 결정하나

    Batch가 N명이면 KV cache 도 N배. 한 사용자 400MB → 1,000 사용자 = 400 GB. 한 GPU의 HBM (~192 GB) 에 안 들어감. 그래서 분산이 필요하고, 그 분산은 다음 모듈에서 보겠지만 pipeline으로는 해결되지 않습니다.

    더 자세히: 왜 K와 V는 캐시하고 Q는 안 하나

    새 토큰을 만들 때, “새 토큰의 쿼리” 가 “이전 모든 토큰의 키” 와 내적합니다. 쿼리는 “이번 토큰” 만의 것이라 캐시할 필요가 없습니다 (한 번 쓰고 끝). 반대로 K와 V는 “이전 토큰들” 의 정보를 미래에도 계속 참조하니 저장 가치가 있는 거죠.

    09 · 미묘함

    “Pipelining is not wise” 의 진짜 의미

    Ilya Sutskever 의 유명한 한 마디. Reiner는 이걸 수식으로 풀어 보여줍니다.

    학습에서의 명백한 문제: pipeline bubble

    학습은 forward 다음에 backward가 따라옵니다. Pipeline 단계가 4개라면:

    • Stage 0가 “micro-batch 1” 의 forward를 끝내고 stage 1로 넘김
    • Stage 0가 “micro-batch 1” 의 backward를 받으려면 모든 stage가 forward를 끝내야
    • 그 사이 stage 0이 잠깐 노는 “bubble” 이 생김

    micro-batch를 잘게 쪼개고 일부 영리한 스케줄링으로 줄일 수는 있지만, 완전히 없애기는 어렵습니다. 이 “idle 시간” 이 바로 학습에서 pipeline이 비싼 이유 중 하나.

    그럼 추론에서는?

    추론은 backward가 없으니 bubble이 없습니다. 그러면 pipeline이 깔끔할까? Reiner는 4가지 차원에서 따져봅니다:

    차원 Pipeline 효과
    Latency (한 토큰 시간) 변화 없음. 어차피 모든 layer를 순서대로 통과 (오히려 hop 지연 추가).
    Compute (총 FLOPs) 변화 없음.
    Weight 메모리 용량 ↓ 1/P 만큼 줄어듦. (P단계로 나눴으니)
    KV cache 메모리/대역폭 변화 없음 ← 결정타.

    왜 KV cache는 P 효과를 못 받나

    P 단계로 나누면 GPU 수가 P배. “가중치는 1/P” 가 분명. 그런데 KV는...?

    “늘어난 GPU를 다 채우려면 batch도 P배 늘려야 한다” 는 게 핵심입니다. 안 그러면 추가 GPU가 놀고 있어서 의미가 없으니까요.

    GPU 한 대당 KV cache 양 KV_per_GPU = (B × P × context × bytes_per_token) / (E × P) = B × context × bytes_per_token / E

    P가 분자, 분모에서 약분되어 사라집니다. 분명히 더 많은 GPU에 펼쳤는데, 각 GPU가 떠받쳐야 하는 KV cache 양은 그대로. 메모리 대역폭 압박도 그대로.

    결정타

    Pipeline은 “가중치 메모리 용량” 만 풀어줍니다. 하지만 frontier serving의 진짜 압박은 가중치 용량이 아니라 KV cache 대역폭입니다. 그 압박을 안 풀어주면서 hop 지연만 추가하니, “not wise” 가 됩니다.

    그래서 frontier lab은

    가능하면 한 scale-up 도메인 안에 다 욱여넣고, pipeline은 최소화. 그래서 Nvidia가 한 랙당 GPU 수를 4배로 끌어올리려고 (Hopper 8 → Blackwell 72 → Rubin 500) 케이블 밀도 같은 물리적 한계와 싸우는 겁니다.

    더 자세히: 그럼 pipeline을 언제는 쓰나

    모델 가중치가 한 scale-up 도메인의 HBM 합보다 클 때는 어쩔 수 없이 pipeline을 씁니다. 예: 5T 파라미터 모델 + 한 랙의 HBM 합 = 13TB → 4 랙 pipeline 필요.

    하지만 한 도메인에 들어가면, pipeline의 “KV 안 풀림 + hop 지연” 비용을 안 내는 게 압도적으로 유리.

    10 · 학습 경제학

    Chinchilla 와 100배 오버트레이닝

    “파라미터 1개당 토큰 20개” 라는 옛 권고는 frontier에서 무시되고 있습니다. 이유는 단순합니다: 학습 비용보다 추론 비용이 훨씬 큽니다.

    Chinchilla 란?

    2022년 DeepMind 논문이 제시한 “compute-optimal” 배합: 파라미터 N 개의 모델은 약 20N 토큰으로 학습할 때 학습 FLOPs 대비 가장 좋은 모델이 됩니다.

    • 70B 모델 → ~1.4T 토큰
    • 100B 모델 → ~2T 토큰

    그런데 frontier 모델은

    추정에 따르면 GPT-5나 Claude 4 같은 모델은 100T+ 토큰으로 학습됐습니다. 100B 파라미터 모델 기준으로 권고치는 2T인데 실제는 ~200T → 100배 over-train.

    이걸 어떻게 합리화 하나? Reiner의 답은 “평생 추론 비용” 입니다.

    비용 균등화 휴리스틱

    두 비용의 합 \(f(x) = 1/x + x\) 는 \(x = 1\) 에서 최소가 됩니다. 비슷하게, “total_cost = train + infer” 처럼 한쪽을 늘리면 다른 쪽이 줄어드는 경우, 두 비용이 같아지는 지점 근처가 자주 최적입니다.

    그래서 frontier는 다음을 비슷하게 맞춥니다:

    3-way 균등화 Pretraining 비용 ≈ RL 비용 ≈ 평생 Inference 비용

    각 비용을 풀어 쓰면:

    • Pretraining: 6 × N_active × D_pre (forward 3 + backward 3)
    • RL: α × N_active × D_RL (α ∈ [2, 6])
    • Inference: 2 × N_active × D_inf (forward만)

    α의 범위가 2~6인 이유: RL의 토큰은 “모델이 직접 생성한 후보들” 이라 상당량은 generation(forward)이고, 일부만 backward. 사실상 “추론에 가까운 학습”.

    실제 숫자로 100배가 나오는 과정

    2025년 frontier 추정값으로:

    • 한 모델 평생 추론 토큰 D_inf ≈ 200T
    • Active params N_active ≈ 100B
    • Inference 비용 = 2 × 100e9 × 200e12 = 4 × 10²⁵ FLOPs
    • 비슷한 규모로 pretrain 비용을 맞추려면 D_pre ≈ (2/6) × 200T ≈ 67T 토큰 — Pope 추정의 “150T” 와 같은 자릿수.
    • Chinchilla 권고는 20 × 100B = 2T 토큰
    결과 D_pre / D_chinchilla = 100T / 2T ≈ 100배
    Reiner의 한 줄

    “각 모델은 자신이 받는 모든 입력에 대한 응답으로 인류의 모든 글의 합을 생성한다.” — 이게 100배 over-training의 정성적 의미입니다.

    더 자세히: 왜 균등화가 최적인가

    비용을 \(C(N, D) = a × N + b/N\) 같은 형태로 쓸 수 있을 때 (한 항은 N에 비례, 다른 항은 1/N에 비례), 미분해서 0 으로 놓으면 두 항이 같아지는 점이 최소가 됩니다.

    학습/추론 trade-off도 비슷합니다: 모델을 작게 만들면 (N ↓) 추론은 싸지지만 같은 quality 유지에는 더 학습이 필요해 학습 비용이 ↑. 두 비용이 같아질 때 합이 최소.

    11 · 가격 리버스 엔지니어링

    Long Context 의 경제학

    “Gemini가 200K에서 가격을 50% 올린다” 는 한 줄에 담긴 정보가 충격적입니다. 이 모듈은 그 한 줄에서 모델 구조까지 거꾸로 푸는 과정을 보여줍니다.

    가격 곡선의 형태

    짧은 컨텍스트에서는 가격이 평탄, 어떤 지점부터 KV fetch가 압도해 비용이 올라감. Gemini가 이 “꺾이는 점” 에 가격 분기를 둠.

    Gemini 2.x는 이렇게 가격을 책정합니다:

    • 컨텍스트 < 200K: 1× 가격
    • 컨텍스트 ≥ 200K: 1.5× 가격

    이건 마케팅이 아니라 그 지점이 메모리 시간 = 계산 시간 의 inflection point 라는 신호입니다.

    거꾸로 푸는 수학

    균형점에서: KV fetch 시간 ≈ 계산 시간.

    균형 조건 (B × L × bytes_per_token) / bandwidth = (B × N_active) / FLOPs

    양변에서 B 약분, 정리:

    L × bytes_per_token = N_active × (bandwidth / FLOPs) = N_active / 300 ← 마법의 숫자!

    L = 200K, N_active = 100B 대입:

    bytes_per_token = 100e9 / (200,000 × 300) ≈ 1,667 bytes ≈ 2 KB

    이게 모듈 8 에서 본 KV cache 공식 “2 × layers × d_head × num_KV_heads” 의 결과와 정확히 일치합니다. Gemini의 가격 한 줄에서 모델의 head 구조까지 거꾸로 풀려 나옵니다.

    또 다른 신호 ① — Prefill vs Decode 가격

    대부분 API 가격에서 출력 토큰이 입력 토큰보다 5-10배 비쌉니다. 왜?

    • Prefill: 사용자 prompt N 토큰을 한 번에 처리. 가중치 한 번 fetch에 N 토큰 모두 처리 가능 → 토큰당 가중치 비용 = 1/N
    • Decode: 한 번에 한 토큰씩 생성. 매 토큰마다 가중치 fetch → 토큰당 가중치 비용 = 1

    비율이 5-10배라는 건 “가중치 fetch가 비용의 5-10배에 달한다” — 즉 frontier 추론은 “심하게 메모리 묶임”.

    또 다른 신호 ② — Cache Hit vs Cache Miss

    많은 API는 “prompt caching” 을 제공합니다. 같은 prefix를 반복 요청하면 두 번째부터는 ~10배 싸집니다.

    • Cache miss = prefill 다시 (forward pass 전체)
    • Cache hit = HBM 에서 KV 끌어오기만
    • 10배 차이 → HBM 접근 비용이 forward의 1/10
    한 점에서 풀려 나온 것들

    “Gemini가 200K에서 가격 50% 올림” 한 줄에서 — KV cache 크기, 모델 head 구조, “심하게 메모리 묶임” 이라는 운영 사실, prompt caching의 경제적 합리성까지 거꾸로 풀려 나옵니다. 이게 Reiner의 “공개 가격으로 frontier lab을 reverse engineering”의 정수입니다.

    왜 200K에서 멈춰 있나

    현재 frontier 모델의 “유효한” 컨텍스트는 1-2년째 100-200K 근처에서 정체. 이유는 메모리 대역폭. Computational cost는 컨텍스트에 선형으로만 늘지만, 메모리 압박이 200K 즈음에서 “batch를 줄여야 함” → “토큰당 비용 ↑” 로 폭발합니다.

    Sparse attention 같은 기법으로 효과적 컨텍스트를 늘릴 수는 있지만 무한정은 아니고, “HBM은 그대로” 라는 물리법칙이 한계. AGI 의 “in-context learning” 이 100M 토큰 컨텍스트를 요구할 거라는 가설이 맞다면, 엄청난 ML 인프라 돌파구가 필요하다는 뜻입니다.

    12 · 종합

    한 장의 그림

    모든 개념이 어떻게 한 의사결정 안에서 맞물리는지 봅시다. 영상 전체가 이 한 장으로 압축됩니다.

    frontier serving 의 “4 숫자 + 1 휴리스틱”

    영상의 모든 결론은 다음에서 나옵니다:

    숫자 출처 결정하는 것
    FLOPs/bandwidth ≈ 300 GPU 데이터시트 최적 batch size, 가격 inflection
    Sparsity 모델 카드 (active/total) batch ≈ 300 × sparsity
    Layer 수, head 차원 모델 카드 bytes/token, 가능한 컨텍스트 길이
    Total params 모델 카드 한 랙으로 가능한지, pipeline 필요 여부
    비용 균등화 휴리스틱 경제적 추론 100배 over-training, RL/inference 분배

    의사결정 흐름

    1. 모델이 한 scale-up 도메인에 들어가나? — 들어가면 expert parallelism + all-to-all 만으로 끝. 안 들어가면 pipeline.
    2. 최적 batch는? — 300 × sparsity. KV 메모리 한도 안에서 그만큼 모음.
    3. 가격은? — 토큰당 가중치 fetch 시간 / batch + 토큰당 KV fetch 시간. Long context 에서는 후자가 빠르게 압도 → inflection 가격.
    4. Pipeline 추가하지 말 것 — Latency 안 줄고, KV 안 줄고, 가중치 메모리만 줆. 한 도메인에 안 들어갈 때만 어쩔 수 없이.
    5. 학습은 100배 over-train — 평생 추론 비용이 학습 비용보다 큰 게 자명하니까, 작은 모델 + 더 학습 = 총 비용 ↓.

    Reiner Pope

    “추론 비용을 모르면 모델 디자인을 이해할 수 없다.” 모델의 거의 모든 결정 — 모델 크기, sparsity, layer 수, head 구성, 학습 토큰 수 — 은 결국 “평생 토큰당 추론 비용을 얼마로 만들 것인가” 라는 한 질문에 의해 결정됩니다. 프론티어 lab들이 외계 어휘로 말하는 것 같지만, 본질은 모듈 2-4의 직관에 다 있습니다.
